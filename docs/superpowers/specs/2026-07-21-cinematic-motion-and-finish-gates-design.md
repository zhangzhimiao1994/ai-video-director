# aibiandao：全链路电影化与反 PPT 验收设计

- 日期：2026-07-21
- 状态：用户已选择 C「全链路电影化」，等待书面规格审核
- 目标技能：`aibiandao`
- 触发案例：`019f75e6-6574-7263-a681-64d0027b7f98`

## 1. 问题定义

现有流程能正确验证故事时长、稳定 ID、连续性、文件存在和编码参数，却允许一条 60 秒、8 个长镜头、全程无声、所有边界均为硬切、人物长期站桩的作品以 `ready: true` 交付。该结果技术上是可播放的粗剪，观感上却是动态海报或 PPT，不属于电影成片。

本次升级把电影感从软建议改为电影化交付的硬门。以下任一情况命中时，作品不得标记为 `fine_cut` 或 `final_master`，也不得使用 `ready: true`、`cinematic_ready: true` 或等价措辞：

- 人物长期保持同一海报式姿态，主要只有粒子、光束、烟雾或背景在动；
- 动作场面缺少准备、释放、命中、反应和后果之间的可见因果；
- 生成镜头与剪辑单元一一绑定，导致 6–10 秒低信息长镜头连续出现；
- 分镜声明了遮挡、匹配切、闪白、声音桥或叠化，时间线却统一降级为无声硬切；
- 电影化精剪或母版没有可验证的环境声、音效、音乐或经过设计的静默；
- 只通过时长、编码和文件探测，就把粗剪当作成片。

## 2. 已确认决策

1. 采用 C「全链路电影化」，不接受只补转场或只改提示词。
2. “不能像 PPT”是电影化交付的硬门，不再是可让步的震撼度建议。
3. 保留现有 A「叙事优先」、B「史诗奇观」、C「表演对白」节奏预设；本设计中的全链路方案不替换这三个预设。
4. 粗剪仍可输出和探测，但必须标记 `creative_ready: false`；它不能因技术探测通过而升级为精剪或母版。
5. 转场必须由动作、遮挡、构图、光线、材质、空间或声音驱动；禁止用随机插件转场掩盖素材僵硬。
6. 无声作品只有在用户明确选择无声形式、静默承担可解释的戏剧作用、且画面运动与表演门通过时，才可能成为电影母版。
7. 先升级 Skill 和验证器，再依据新规则制作当前项目的新分镜、v003 挽救版和 v004 重生版；旧版本全部保留。
8. 内容一致性与人物一致性继续作为硬门，并升级为可追踪的跨剧本、分镜、提示词、任务和成片证据链。
9. 每个角色必须先批准唯一 `identity_profile_id`；进入可执行生成前，还必须锁定所选平台的模型家族、明确版本、主体绑定方式和参考资产集合。
10. 同一平台内不得为同一角色静默更换模型版本、身份配置、参考图集合或主体绑定方式。确需迁移时，必须先生成对照测试并重新通过人物一致性审阅。

## 3. 目标与非目标

### 3.1 目标

- 让动作、表演、摄影机、环境和剪辑共同产生电影运动，而不是只让特效层运动。
- 把一次主要攻击拆成可剪辑的因果覆盖：准备或意图、释放、命中、对手反应、环境后果、恢复或反转。
- 让素材生成时长与最终剪辑单元时长解耦，允许从一个素材中选择多个短区间，也允许一个故事动作由多个独立镜头覆盖。
- 逐边界兑现分镜中的转场与声音设计，并对任何降级留下可审计理由。
- 对计划和实际成片同时运行反 PPT 检查；没有视觉审阅或工具证据时不得声称电影验收通过。
- 锁定内容事件、人物动机和结局，并让每个角色在所有画幅、镜头和平台任务中继承同一批准身份配置。
- 把角色视觉身份与平台模型锁分开管理：视觉身份跨平台共享，模型与绑定方法在每个平台内部固定。
- 保持旧十对象制作包、现有三道创意审批门和操作授权边界兼容。

### 3.2 非目标

- 不把电影感简化为固定数量的炫技转场、持续运镜或每秒切镜。
- 不强迫对白戏、悬疑停顿或有意长镜头使用动作片节奏。
- 不使用随机缩放、无动机抖动、滥用速度坡度或全片闪白制造虚假动感。
- 不用后期剪辑伪造素材中不存在的关键表演、肢体接触、命中或角色反应。
- 不覆盖、重命名或删除现有 v001、v002 及源素材。

## 4. 总体架构

电影化链路调整为：

`故事因果 → 动作覆盖图 → 运动分镜 → 平台提示词 → 素材运动审阅 → 剪辑节奏与动机转场 → 声音/色彩 → 反 PPT 验收 → 电影母版`

新增的是现有对象中的字段和验证，不创建与 Canon 竞争的平行故事对象。

### 4.1 内容与角色双锁

内容一致性使用现有稳定事件、scene、beat、shot 与 boundary state 继续追踪，并增加电影化删除/顺序审计。人物一致性使用两层锁：

1. **跨平台视觉身份锁**：每个 `continuity_bible.characters[]` 拥有唯一 `identity_profile`，至少包含 `identity_profile_id`、批准状态、脸部锚点、体态/比例锚点、发型锚点、不可变配饰、法术或能力视觉签名、批准参考资产 ID 和禁止漂移项。
2. **平台内生成模型锁**：每个包含人物的电影化 job 使用 `character_model_bindings`，逐角色记录 `character_id`、`identity_profile_id`、`model_family`、`model_version`、`identity_binding_method`、`reference_input_ids` 与 `lock_status`。

同一角色在同一 `model_family` 下的模型版本、身份配置、绑定方式和身份参考集合必须完全一致。画幅可以改变构图与调度，但不得改变人物模型锁。模型或参考链未知时，job 只能为 `blocked` 或 `non_executable`；不得用 `approved` 掩盖未确定的人物模型。

角色模型迁移只能通过显式 `migration_test`：旧配置与新配置使用同一中性表情、同一光线、同一景别和同一服装生成对照，检查脸部锚点、比例、发型、配饰与肤色。对照未通过时保持旧锁，或阻断受影响任务。

### 4.2 动作覆盖图

对动作、斗法、追逐、灾难和奇观 beat，`storyboard` 必须使用 `coverage_role` 标记每个镜头的职责：

- `setup`：建立人物、目标、空间或危险；
- `anticipation`：蓄力、预判、视线、重心或环境先兆；
- `action`：角色完成可见主动作；
- `impact`：接触、命中、破裂或力量传递；
- `reaction`：对手、群体或施法者受到结果；
- `consequence`：环境、关系、目标或世界状态改变；
- `transition`：以可见或可听机制连接新空间或新节拍；
- `aftermath`：恢复、代价、余韵或新的危险。

每个主要攻击事件至少拥有 `action`、`reaction`、`consequence` 三类可见证据。`impact` 可以与其中一类合并，但不得只用特效光束替代角色反应和物理后果。若模型能力不足以在一镜完成，必须拆镜或选择低风险覆盖，不得把所有动作压进一个长镜头。

### 4.3 逐镜运动契约

每个电影化 active shot 增加 `kinetic_profile`：

```json
{
  "subject_motion": "角色从防御重心转为挥袖释放",
  "performance_change": "冷笑在命中反噬时消失",
  "camera_motion": "短促前推在释放瞬间结束",
  "environment_motion": "冲击风先推衣发，再撕开云层",
  "motion_layers_required": 2,
  "intentional_hold": false,
  "hold_reason": null,
  "acceptance_evidence": "动作、衣发受力和环境后果在连续帧中均可辨"
}
```

动作或奇观镜头必须至少有两层有效运动，层级来自主体、表演变化、摄影机或环境。纯粒子漂动不单独构成有效主体运动。`intentional_hold: true` 只用于明确的悬念、凝视、信息读取或余韵，并必须记录开始、结束、时长和戏剧作用。

### 4.4 景别与覆盖变化

电影化动作段不能持续使用同一人物全身正面构图。镜头计划必须审计：

- 建立镜、人物动作镜、命中或插入镜、反应镜和结果镜是否存在；
- 相邻镜头是否改变信息、视角、景别、空间关系或情绪，而不是只更换背景特效；
- 是否连续出现海报式正面站姿；
- 英雄镜头和长停留是否由前序节奏挣得，而非填满目标时长。

镜头数量与平均镜头长度按题材和节奏预设评估，不使用适用于所有作品的单一数字。对于 60 秒动作奇观，默认审查区间为 18–30 个 active shot 或等价的 18–30 个有效 edit unit；超出区间不自动失败，但必须提供节奏理由和视觉证据。连续三个 5 秒以上的低运动镜头直接触发 PPT 风险。

### 4.5 生成素材与剪辑单元解耦

`shot_id` 继续标识故事与生成镜头，`edit_unit_id` 标识最终时间线选择。一个素材可以通过不同合法 source in/out 形成多个 edit unit；一个故事动作也可以由多个 shot 覆盖。系统不得默认把每条 6–10 秒生成素材原长放入时间线。

每个 edit unit 必须说明：

- 为什么从该素材的这个区间开始和结束；
- 它承担哪个 `coverage_role`；
- 切点由动作、视线、遮挡、构图、声音还是故事结果触发；
- 是否需要速度变化、稳定、重构或视觉效果；
- 该处理是否掩盖素材缺陷；若是，则阻断电影母版并返回素材重生。

## 5. 动机转场硬门

每个镜头边界必须包含结构化 `transition_contract`：

```json
{
  "type": "occlusion_match_cut",
  "visual_precondition": "云白在前镜末尾覆盖全幅",
  "incoming_match": "后镜从同亮度雷光中显影",
  "duration_frames": 4,
  "audio_bridge_cue_id": "CUE-THUNDER-01",
  "story_reason": "把云妖出招直接交给雷妖反击",
  "fallback": "在雷声 J-cut 上硬切，不使用无动机叠化"
}
```

剪辑计划必须继承获批 storyboard 的转场。若实际素材不满足视觉前提，只能使用已声明 fallback、退回镜头重生或提交新的 change summary；不得把所有边界静默归一化为 `hard_cut`。硬切本身允许，但必须有动作、构图或声音理由。

## 6. 声音成片硬门

电影化 `fine_cut` 和 `final_master` 必须具有可验证的声音结构：

- ambience 建立空间连续性；
- SFX 证明动作、材质、命中和世界后果；
- music 或经过设计的无音乐结构组织 build、drop、高潮与余韵；
- silence 是有起止、有目的的 cue，不等于缺少音轨；
- J-cut、L-cut 或同步 cue 参与关键转场。

最终文件没有音频流时，默认阻断电影母版。只有用户明确批准的无声电影形式可以豁免音频流，但仍必须通过表演、运动、节奏和转场验收，并记录 `silent_form_authorization`。

## 7. 反 PPT 验收

`edit_master_plan` 增加 `cinematic_validation`，至少包含：

- `declared_mode` 与题材/节奏预设；
- `content_consistency` 与锁定事件映射；
- `character_identity_integrity`、角色身份配置和平台模型锁审计；
- `action_reaction_coverage`；
- `kinetic_profile_audit`；
- `shot_scale_and_composition_variety`；
- `transition_fulfillment`；
- `audio_presence_and_structure`；
- `static_hold_audit`；
- `source_motion_review`；
- `ppt_risk_flags`；
- `cinematic_ready`、证据引用和失败返回点。

验收分两层：

1. **计划验收**：检查字段、时间线、覆盖角色、切点、转场、声音和禁项是否完整。
2. **成片验收**：使用 ffprobe/FFmpeg 等工具证据检查流、时长、切点和帧变化，并通过抽帧接触表或视觉审阅确认人物动作、表演变化、重复姿态和实际转场。

像素差异只能证明画面发生变化，不能证明人物在表演。粒子闪烁、光束变色或背景流动不能单独让 `source_motion_review` 通过。

### 7.1 直接失败条件

电影化精剪或母版出现下列任一项，`cinematic_ready` 必须为 `false`：

- 无视觉审阅证据或成片探测证据；
- 缺少必要的 action/reaction/consequence 覆盖；
- 锁定事件缺失、重排、动机改变、结局改变，或内容审计没有证据；
- 人物脸部锚点、体态比例、发型、服装、配饰或法术签名漂移；
- 同一平台内同一角色的模型版本、身份配置、绑定方式或参考集合发生未批准变化；
- 人物模型或身份参考链仍为 unresolved，却把 job 标记为 `approved`；
- 人物持续站桩而提示词或报告只强调特效运动；
- storyboard 转场未兑现且没有获批降级记录；
- 音频流缺失且没有无声形式授权；
- 多个长镜头仅依赖轻微推拉、呼吸、眨眼或衣摆漂动填充；
- `ppt_risk_flags` 非空却仍设置 `ready: true`；
- 把 `rough_cut` 改名或复制为 `fine_cut` / `final_master`。

失败必须返回最早可修复位置：故事因果、动作覆盖、分镜、提示词、素材选择、素材重生、时间线、声音或交付标记。禁止只在结尾加转场包后重新申报通过。

## 8. 验证器和状态机

`scripts/validate_edit_plan.py` 增加电影化验收模式。推荐接口：

```text
python scripts/validate_edit_plan.py PLAN.json --require-cinematic
python scripts/validate_edit_plan.py PLAN.json --require-final --require-cinematic
```

兼容规则：

- 旧计划不带 `cinematic_validation` 时，普通粗剪验证保持兼容；
- 声明电影化 fine cut 或 final master 时，缺少该对象直接失败；
- `rendered` 只说明工具已经输出文件，不等于 `cinematic_ready`；
- aggregate `plan_status` 不得把技术成功覆盖创意失败；
- `ready: true` 的电影化交付必须同时满足技术、版权、连续性和电影验收。

## 9. 当前项目重制路线

### 9.1 v003：现有素材挽救版

目标是证明现有 8 条素材能被挽救到什么程度，不把它冒充彻底重生：

- 保留 v001/v002，只创建新版本目录；
- 先审计八条素材中的人物身份；发生明显换脸、年龄、体态、发型或服装漂移的素材不得进入挽救时间线；
- 从现有素材选择动作和特效变化最明显的短区间；
- 生成约 18–24 个 edit unit，使用动作切、构图匹配、云雾/雷光/水墙/岩尘遮挡、声音桥和受控速度变化；
- 新建音乐、环境声、材质音效、冲击音和两秒设计静默；
- 目标时长暂定 45–50 秒，正式执行前在剪辑 dry-run 中锁定；
- 若关键出招、反应或世界倒转仍无法从素材中读出，v003 只能标记为 `fine_cut_candidate` 或 `blocked_for_regeneration`，不得强行通过电影母版。

### 9.2 v004：重新生成的电影版

v004 从新分镜开始，不沿用“8 条素材原长拼成 60 秒”的结构：

- 保留四妖、四界倒转、无人获胜和 60 秒故事事实；
- 为 C01-C04 分别批准唯一身份配置，并为实际使用的平台锁定模型版本、主体绑定方式和身份参考集合；
- 重新设计约 20–26 个 active shot，覆盖建立、凝视、蓄力、出招、对手反应、命中、环境后果、共同误判、世界倒转和余韵；
- 每次主要斗法至少有施法、反应和世界后果三个可剪辑证据；
- 生成提示词明确身体重心、手臂路径、视线变化、衣发受力、冲击反作用和环境响应；
- 关键镜头准备短动作主镜和低风险反应/结果 fallback；
- 完成正式声音、动机转场、双画幅重构和电影化验收后，才允许成为 `final_master`。

## 10. RED-GREEN-REFACTOR 测试

### 10.1 RED：固定当前失败

新增一个基于本次事故的脱敏 fixture，证明旧行为会错误通过：

- 60 秒、8 个 active shot，平均 7.5 秒；
- 所有边界硬切；
- `audio_tracks` 为空，实际文件无音频流；
- storyboard 声明了多种转场，时间线未兑现；
- 多数 performance 为静止、抬眼、呼吸或身体不位移；
- delivery 是 `rough_cut`，但报告错误设置可作为电影成片使用。

在修改生产规则前，新电影验收测试必须因上述原因失败，并输出可读的 PPT 风险错误。

### 10.2 GREEN：最小实现

- 文档要求动作覆盖、逐镜运动、转场、声音和反 PPT 验收；
- 验证器拒绝无声硬切动作母版；
- 有明确 action/reaction/consequence、运动证据、动机转场、声音结构和视觉审阅证据的 fixture 通过；
- `rough_cut` 技术渲染成功但 `creative_ready: false` 的 fixture 合法且不能冒充母版；
- 明确授权的无声形式按专门规则验证。

### 10.3 REFACTOR：防止误伤

- 对白戏的有意停顿、悬疑长镜和静默段落不能被动作片阈值误杀；
- 普通非电影化剪辑仍可使用旧验证路径；
- 硬切有叙事或声音动机时不被错误禁止；
- 粒子运动不能冒充人物表演；
- 不通过堆砌字段或伪造 `review_status` 绕过证据要求。

## 11. 文件影响

预计修改：

- `SKILL.md`：加入电影化成片硬门、失败回退和反模式；
- `references/cinematic-directing.md`：动作覆盖、运动层和景别变化；
- `references/story-directing.md`：动作因果和声音节拍验收；
- `references/continuity-storyboard.md`：`coverage_role`、`kinetic_profile`、转场契约；
- `references/prompt-compiler.md`：身体动作、反作用、环境响应与静态海报黑名单；
- `references/editing-finish.md`：素材/剪辑解耦、转场兑现、声音硬门和 `cinematic_validation`；
- `references/output-contract.md`：电影化验证字段和状态语义；
- `scripts/validate_edit_plan.py`：`--require-cinematic` 与反 PPT 结构验证；
- `scripts/validate_package.py`：内容一致性、角色身份配置、job 人物模型锁和跨任务漂移验证；
- `scripts/build_edit_bundle.py`：电影验收报告、接触表/探测证据和交付阻断；
- `tests/` 与 fixtures：本次事故回归、电影通过、粗剪非成片、无声授权和题材阈值测试；
- 制作包测试还必须覆盖未知身份配置、人物引用缺失、模型版本漂移、参考集合漂移和未通过迁移测试；
- `test-prompts.json` 与 `results.tsv`：加入本次典型失败和升级评估记录。

不新增强制第三方 Python 依赖。FFmpeg/ffprobe 不可用时，可以生成计划和施工包，但实际电影母版验收保持 `blocked` 或 `manual_review_required`。

## 12. 成功标准

1. 本次 8 镜头、60 秒、无声、全硬切案例在电影化模式下必定失败，错误明确指出 PPT 风险。
2. 一个文件“能播放”不再等于“电影成片 ready”。
3. 动作奇观必须提供 action/reaction/consequence 和逐镜运动证据。
4. 内容锁定事件、人物动机、因果顺序和结局在所有下游对象中保持一致。
5. 每个出镜角色都拥有批准身份配置；同一平台内的模型版本、主体绑定方式和参考资产集合不会漂移。
6. 分镜中的动机转场不能在剪辑阶段被静默抹除。
7. 电影化精剪和母版必须有声音结构，或拥有明确的无声形式授权。
8. 实际成片必须经过工具探测和视觉审阅；只审 JSON 不能通过电影验收。
9. v003 明确区分可挽救与必须重生的部分，不能靠转场掩盖素材站桩或人物漂移。
10. v004 使用新分镜、锁定人物模型和新素材完成 60 秒电影化版本，并保留所有旧版本。
11. 现有非电影化、剧本、分镜、提示词、平台任务和普通粗剪工作流无回归。
12. Skill、文档、验证器、构建器、fixtures 和已安装版本通过完整回归后才允许发布。
