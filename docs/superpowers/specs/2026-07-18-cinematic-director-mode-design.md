# AI Video Director：通用电影化编导模式设计

- 日期：2026-07-18
- 状态：已获用户批准
- 目标仓库：`ai-video-director`
- 目标版本：下一次功能迭代

## 1. 背景

现有 AI 视频生成容易出现一种“说不清哪里错，但整体很奇怪”的感觉。问题通常不只来自单条提示词，而是来自上游缺失：剧情因果没有锁定、角色和场景状态没有统一事实源、镜头之间缺乏依赖关系、平台能力差异被混入导演逻辑，以及为了追求“震撼”同时堆叠过多动作、运镜和特效。

本次设计参考用户提供的抖音《神性游戏》第 41 集。该集的有效电影感主要来自镜头尺度变化、前景遮挡、景深、体积光、冷暖对比、角色与服装一致性、反应镜头、世界建立镜头和对白节奏，而不是持续的高强度特效。第 41 集页面标注了商汤 Seko 支持；同系列其他集数存在 Seedance 创作标注。因此本模式提炼的是可迁移的电影语言，不绑定或复刻某个具体作品、IP 或生成平台。

参考：

- [用户提供的《神性游戏》第 41 集页面](https://www.douyin.com/jingxuan/search/%E7%A5%9E%E6%80%A7%E6%B8%B8%E6%88%8F41?aid=b7e0adcb-7e13-46b0-8020-9de48c069220&modal_id=7663111383977839914&type=general)
- [Doubao Seedance 2.0 系列提示词指南](https://www.volcengine.com/docs/82379/2222480?lang=zh)
- [HappyHorse 视频模型能力](https://www.alibabacloud.com/help/en/model-studio/video-generate-edit-model)
- [HappyHorse 文生视频 API](https://www.alibabacloud.com/help/en/model-studio/happyhorse-text-to-video-api-reference)

## 2. 已确认的产品决策

1. 首版生成 30–60 秒电影级片段或预告，不直接承担 3–5 分钟完整单集。
2. 采用多平台架构，首批面向可灵、Seedance、HappyHorse，并允许后续增加平台。
3. 支持两种首版输入：
   - 用户有完整剧本：进行电影化压缩与改编。
   - 用户只有故事灵感：自动补全剧情、角色、分镜与提示词。
4. 为后续“小说改编”预留入口，但首版不实现章节级长篇改编。
5. 电影化模式为通用能力，不固定为暗黑西幻；题材通过视觉预设切换。
6. 同时输出 16:9 电影母版与 9:16 竖屏重构版。
7. 角色一致性和剧情可理解性是硬门槛；镜头震撼度是可让步的软指标。
8. 采用分层 AI 编导流水线，关键镜头可按需增加多角色评审。
9. 默认镜头节奏为 A“叙事优先电影化”；B“史诗奇观”和 C“表演对白”为可选预设。

## 3. 目标与非目标

### 3.1 目标

- 把故事、剧本或未来的小说输入转换为统一、可验证的电影化生产包。
- 在生成平台之前锁定剧情事实、角色身份和跨镜头状态。
- 让每个镜头都有明确的叙事功能、状态变化和连续性依赖。
- 为不同平台生成适配提示词和任务清单，同时避免虚构参数。
- 让横屏和竖屏共享剧情事实，但拥有各自的构图与调度。
- 在不牺牲故事和角色的前提下提升构图、光影、节奏、运镜和声音设计。

### 3.2 非目标

- 首版不直接编排 3–5 分钟连续剧集。
- 首版不执行真实 API 请求、管理密钥或承担素材永久存储。
- 不承诺跨平台输出完全相同的像素结果。
- 不复刻《神性游戏》的具体角色、台词、镜头或受保护资产。
- 不为未核验的平台接口生成看似可运行的伪 JSON。
- 不把“更多特效、更多运镜”当作电影感的唯一指标。

## 4. 总体架构

主流程：

`输入路由 → 剧情核心 → 资产圣经 → 镜头关系图 → 连续性质检 → 电影化增强 → 双画幅重构 → 平台编译器 → 交付包`

### 4.1 输入路由器

输入路由器识别以下模式：

- `concept_mode`：一句话、主题、人物设定或故事灵感。
- `screenplay_mode`：完整或部分剧本。
- `novel_mode_reserved`：未来的小说、章节或长篇文本入口。

所有模式最终转换为同一种 `Canon Spec`。小说入口未来只需要增加上游的章节拆解、人物关系抽取、事件选择和集数规划，不改动后续导演核心。

### 4.2 剧情核心

剧情核心先生成或压缩出：

- 一句话故事；
- 主角目标；
- 阻碍与代价；
- 因果链；
- 关键反转；
- 高潮选择；
- 结尾余韵或悬念。

默认采用自适应六段节奏：

`钩子 → 建立目标 → 冲突升级 → 信息反转 → 高潮选择 → 余韵/悬念`

六段结构不是固定秒表。对白戏允许放慢，动作戏允许压缩，但每个段落必须承担可说明的剧情作用。

### 4.3 Canon Spec 与资产圣经

`Canon Spec` 是全项目唯一事实源，至少包含：

- 角色 ID、身份、关系和动机；
- 面部锚点、体型、发型、服装、材质与标志性道具；
- 场景空间、时间、天气、光源和色彩状态；
- 人物能力、伤势、污渍、服装变化和情绪状态；
- 事件发生顺序；
- 禁止擅自新增、删除或改变的事实；
- 用户提供素材与生成素材的来源标识。

资产圣经从 `Canon Spec` 派生角色卡、服装卡、道具卡、场景卡、色彩脚本和参考图生成提示词。

### 4.4 Shot Graph

镜头以有向关系图组织，而不是互不相关的提示词列表。每个镜头至少包含：

```yaml
shot_id: S01
duration_seconds: 5
narrative_function: establish_goal
characters: [CHAR_A]
state_before: {}
state_after: {}
location_and_screen_direction: {}
composition_16x9: {}
recomposition_9x16: {}
camera_and_blocking: {}
lighting_and_atmosphere: {}
dialogue_and_audio: {}
required_reference_assets: []
continuity_dependencies: []
negative_constraints: []
platform_capability_needs: []
```

镜头依赖覆盖：

- 人物进入与离开位置；
- 视线与轴线方向；
- 动作的开始和完成状态；
- 道具归属和位置；
- 服装、伤势、光线和天气变化；
- 上一镜尾帧到下一镜首帧的承接；
- 对白、环境声和音乐节拍。

修改上游镜头时，系统必须指出受影响的下游镜头。

## 5. 质量门槛

### 5.1 硬门槛

以下任一不合格时，不能生成最终平台提示词包：

- `narrative_clarity`：观众能否理解人物、目标、阻碍和结果。
- `continuity_integrity`：角色、服装、道具、空间和人物状态是否连续。

剧情失败时返回 Beat Map 修复；连续性失败时冻结已通过的剧情，只修复问题镜头和相关下游镜头。

### 5.2 软指标

- `cinematic_impact`：尺度、构图、光影、节奏和声音是否具有电影感。
- `spectacle_density`：奇观镜头的密度是否匹配题材和预算。

软指标只给出改进建议，不阻止交付。不得为提升软指标而破坏硬门槛。

## 6. 镜头节奏预设

### 6.1 A：叙事优先电影化（默认）

- 30–60 秒通常规划 7–12 个镜头，允许按故事复杂度调整。
- 使用世界建立镜头说明尺度与空间。
- 使用近景、双人镜头、反应镜头和道具插入讲清人物关系。
- 只设置 1–2 个高投入英雄镜头。
- 结尾使用结果镜头、情绪余韵或悬念镜头。
- 炫技镜头必须同时承担叙事功能。

### 6.2 B：史诗奇观（可选）

- 增加大全景、复杂空间、动态构图、特效和镜头运动。
- 仍受剧情和连续性硬门槛约束。
- 当参考素材不足或角色锚点不稳定时，自动降级为 A。

### 6.3 C：表演对白（可选）

- 增加近景、特写、双人覆盖和反应镜头。
- 强调微表情、停顿、视线和台词节奏。
- 减少复杂场景切换和大规模特效。

### 6.4 反“奇怪感”约束

默认每个镜头只保留：

- 一个主要动作；
- 一个主要运镜；
- 一个主要情绪转折；
- 一个清晰的视觉关注点。

系统应检测并拆分相互冲突的动作、运镜、景别、光线和特效描述。

## 7. 双画幅策略

16:9 是电影母版，但 9:16 是独立重构，不是机械裁切。

### 7.1 16:9 母版

- 使用横向人物调度；
- 强调前、中、后景；
- 允许更大的环境尺度和多人空间关系；
- 提供竖屏安全区参考，但不因此牺牲横屏构图。

### 7.2 9:16 重构

- 重新安排人物站位和视线；
- 使用纵向景深、上下层次和高度关系；
- 预留字幕、UI 和平台安全区；
- 必要时把横屏多人镜头拆成主镜头与反应镜头；
- 无法安全重构时标记为“必须独立生成”，并输出新的竖屏提示词。

两种画幅必须共享同一 `Canon Spec` 和剧情状态。

## 8. 提示词编译

提示词分成三层：

1. `global_lock_block`：角色、世界观、材质、色彩、时代、禁止项。
2. `shot_direction_block`：表演、调度、构图、镜头、光影、声音和时间进程。
3. `platform_compile_block`：平台支持的素材引用方式、时长、比例、清晰度、任务类型和负面提示。

统一镜头规范必须先于平台提示词存在。切换平台时只重新编译第三层，不得重写剧情和角色事实。

## 9. 平台能力注册表与适配器

每个平台适配器包含：

```yaml
platform:
adapter_version:
verified_on:
official_docs_url:
supported_task_types: []
supported_ratios: []
supported_durations: []
reference_inputs: []
audio_support:
async_job_model:
result_expiry:
known_limitations: []
```

适配原则：

- 优先选择能利用参考图、首帧或参考视频的任务类型，以提高角色一致性。
- 平台支持首尾帧时，可用于镜头承接；不支持时改用共享首帧或剪辑过渡。
- 平台参数只有在官方文档核验后才能进入可执行任务清单。
- 无法核验的平台输出人工可读的操作清单，并明确标记 `manual_only`。
- API 密钥只使用环境变量占位符，禁止写入 Skill、示例或测试。

首批适配范围：

- 可灵：保留现有适配能力，实施时重新核验官方文档和当前模型。
- Seedance：使用独立提示词编译规则，依据火山引擎官方指南核验。
- HappyHorse：覆盖文生视频、首帧图生视频和参考图生视频；任务与结果处理遵循官方异步接口说明。
- 未知平台：使用通用手工适配器，不猜测字段。

## 10. 关键镜头评审

默认不对全部镜头运行多角色反复评审。只有以下镜头可以触发额外评审：

- 世界建立镜头；
- 剧情反转镜头；
- 高潮选择或英雄镜头；
- 双画幅重构困难的多人镜头；
- 连续性质检重复失败的镜头。

评审角色包括编剧、导演、摄影和连续性监督。评审输出必须是局部修改建议，不能擅自改变已锁定剧情。

## 11. 异常与降级处理

### 11.1 剧情不清

退回 Beat Map，修复人物目标、因果、反转或结果。不得仅靠增加旁白掩盖问题。

### 11.2 角色或状态漂移

冻结剧情和已通过镜头，重写问题镜头的资产引用、状态字段和负面约束，并重新检查依赖镜头。

### 11.3 素材不足

先输出角色三视图、服装、场景和道具参考图提示词。关键资产未补齐前，不进入正式镜头提示词阶段。

### 11.4 平台能力不足

按以下顺序降级：

`参考视频 → 多参考图 → 首帧/尾帧 → 单首帧 → 纯文本 → 手工任务清单`

每次降级都要说明对一致性、成本和质量的影响。

### 11.5 平台任务失败

任务清单应保留请求摘要、素材版本和任务 ID，并建议幂等轮询或重试，避免重复提交。临时结果链接必须提示及时下载归档。

### 11.6 双画幅冲突

如果主体、道具或空间关系无法同时满足两个画幅，则输出独立竖屏镜头，不执行强制裁切。

## 12. 交付结构

完整模式输出：

```text
00-director-brief/
01-screenplay/
02-canon-and-asset-bible/
03-storyboard-and-shot-graph/
04-keyframe-prompts/
05-platform-prompt-packs/
06-16x9-and-9x16-versions/
07-continuity-and-story-qc/
```

最小输出仍必须包含：

- 导演摘要；
- 30–60 秒剧本或节奏图；
- Canon Spec；
- 逐镜头分镜；
- 角色与剧情质检结果；
- 至少一个平台的提示词或手工任务清单。

## 13. 测试设计

实现必须遵循测试驱动流程：先编写能够证明新模式尚不存在或不满足设计的失败测试，再修改 Skill，最后运行完整回归测试。

### 13.1 场景测试

- 故事灵感自动扩写；
- 完整剧本压缩改编；
- 双人对白；
- 多角色复杂场景；
- 角色换装、受伤、持有和丢失道具；
- A/B/C 节奏预设；
- 16:9 与 9:16 重构；
- 可灵、Seedance、HappyHorse 平台编译；
- 未知平台安全降级；
- 小说入口占位兼容；
- 旧版调用方式回归。

### 13.2 不变量测试

- 只切换平台时，剧情事实、角色 ID 和状态不变。
- 只切换画幅时，剧情事实和对白含义不变。
- 只切换 A/B/C 时，剧情事件顺序不变。
- 修复单镜头时，不应无故重写无依赖的镜头。

### 13.3 结构验证

验证器应检查：

- 必要输出目录或章节是否存在；
- Shot ID 和依赖是否可解析；
- 角色、道具和场景引用是否存在于 Canon Spec；
- `state_before` 与上游 `state_after` 是否冲突；
- 双画幅是否都存在，或是否明确标记无法重构；
- 未核验平台是否错误输出可执行 JSON；
- 剧情与连续性硬门槛是否通过。

### 13.4 回归要求

- 现有 43 项测试必须继续通过。
- 新测试不得降低旧版输出契约。
- Skill 文档、参考文件、验证器和测试样例必须保持一致。

## 14. 预期文件影响

实施计划可以调整具体拆分，但预计涉及：

- `SKILL.md`：增加电影化模式路由和使用说明。
- `references/story-directing.md`：增加两种输入入口与六段节奏。
- `references/continuity-storyboard.md`：增加 Canon Spec、Shot Graph 和状态依赖。
- `references/prompt-compiler.md`：增加三层提示词与反奇怪感规则。
- `references/model-adapters.md`：增加能力注册表和首批平台适配。
- `references/output-contract.md`：增加双画幅与交付结构。
- `scripts/validate_package.py`：增加新结构和不变量验证。
- `test-prompts.json`、`tests/test_validate_package.py`：增加失败用例、回归用例和平台降级用例。

## 15. 验收标准

设计完成实现后，应满足：

1. 一句故事灵感可以得到完整的 30–60 秒生产包。
2. 完整剧本可以在不改变核心因果和人物动机的前提下被压缩改编。
3. A 为默认预设，B/C 可显式选择。
4. 剧情或连续性失败时不生成最终平台提示词包。
5. 16:9 和 9:16 共享剧情事实，但具有各自构图。
6. 切换平台不改变剧情、角色和镜头状态。
7. 可灵、Seedance、HappyHorse 输出来自已核验能力；未知平台安全降级。
8. 现有 43 项测试和新增测试全部通过。
9. 输出能明确告诉用户先生成哪些资产、再生成哪些镜头，以及哪些镜头需要重做。

## 16. 后续扩展

首版稳定后，可继续增加：

- 小说章节拆解与人物关系图；
- 3–5 分钟完整单集规划；
- 连续剧跨集 Canon；
- 成片反向质检与自动重生成建议；
- 成本、延迟和失败率驱动的平台自动路由；
- 更多题材视觉预设与平台适配器。
